### Как происходит процесс аутентификации:
 - Пользователь предоставляет учетные данные (логин/пароль, отпечаток, код из SMS)
 - Сервер проверяет данные в системе
 - Если данные верны -> создается сессия / выдается токен
 - Пользователь считается аутентифицированным
 - При последующих запросах проверяется сессия

### Методы аутентификации
1. По фактору знания (Что вы знания?)
 - Пароль
 - PIN-код
 - Секретный вопрос
 - Графический ключ
2. По фактору владения (Что у вас есть?)
 - SMS-код
 - Email-код
 - TOTP
 - Аппаратный токен
 - Push-уведомление
3. По фактору биометрии (Кто вы?)
 - Отпечаток пальца
 - Face ID / Распознавание лица
 - Радужная оболочка глаза
 - Голос
 - Поведенческая биометрия
4. Многофункциональная аутентификация (MFA/2FA)
   Комбинация двух и более факторов для повышения безопасности

### Техническая реализация
1. Сессии на сервере (Классический подход)
 Плюсы:
 - Простота реализации
 - Легко отозвать
 - Данные хранятся на сервере
 Минусы:
 - Требуется хранение сессий (память/БД/Redis)
 - Сложнее масштабировать (sticky sessions или общее хранилище)
2. Токены (JWT, OAuth) (Stateless подход)
 Плюсы:
 - Масштабируемость (нет состояния на сервере)
 - Подходит для мобильных приложений и микросервисов
 - Кросс-доменная аутентификация
 Минусы:
 - Сложнее отозвать токен до истечения срока
 - Токен может быть больше, чем ID сессии
 - Требует тщательной настройки безопасности
3. OAuth 2.0 / OpenID Connect (Делегированная аутентификации)
   Позволяет пооользователю войти через сторонний провайдер (Google, GitHub, VK)

### Лучшие практики
 - Хеширование паролей
 - Защита от брутфорса. Ограничение попыток входа, CAPTCHA, задержки
 - HTTPs всегда
 - Защита сессий. `seession_regenerate_id()`, `HttpOnly` cookie, SameSite
 - Валидация ввода. Санитация логина/пароля, защита от инъекций
 - Логирование. Запись успешных и неудачных попыток ввода
 - Уведомление. Оповещение о входе с нового устройства/локации
 - Регулярная ротация. Принудительная смена паролей для критических систем

### Жизненный цикл аутентификации
1. Регистрация
  - Пользователь создает создает учетную запись
  - Проверка уникальности логина/email
  - Хеширование пароля
  - Подтвержение через email/SMS (опционально)
2. Вход (Login)
  - Пользователь предоставляет учетные данные
  - Сервер проверяет данные
  - При успехе: создание сессии / выдача токена
  - При неудаче: логирование, защита от брутфорса
3. Поддержание сессии
  - Проверка аутентификации при каждом запросе
  - Продление сессии при активности
  - Обновление токена (refresh token)
  - Мониторинг подозрительной активности
4. Выход (Logout)
  - Уничтожение сессии на сервере
  - Отзыв токена (blacklist)
  - Очистка cookies на клиенте
  - Логирование выхода
5. Восстановление
  - Запрос на сброс пароля
  - Генерация одноразового токена
  - Отправка ссылки на почту
  - Установка нового пароля

### Чек-лист безопасной аутентификации
 - Пароли хранятся только в хешированном виде (bcrypt/argon2)
 - Используется HTTPS для всех запросов аутентификации
 - Реализована защита от брутфорса (rate limiting, CAPTCHA)
 - Сессии используют HttpOnly + Secure + SameSite куки
 - Реализован механизм восстановления пароля
 - Ведётся логирование попыток входа
 - Пользователи получают уведомления о новых входах
 - Поддерживается многофакторная аутентификация (опционально)
 - Токены/сессии имеют разумное время жизни
 - Реализован безопасный механизм выхода (logout)

Check list:
-[] Токены хранятся в HttpOnly + Secure cookie
-[] Используется HTTPS в продакшене
-[] Access Token живет не более 15 минут
-[] Реализован механизм Refresh Token
-[] Секретный ключ хранится в переменных окружения
-[] На сервере включена строгая валидация подписи и алгоритма
-[] В токене нет чувствительных данных
-[] Настроены заголовки CORS (если домены разные)
-[] Реализована возможность отзыва токенов (logout)
